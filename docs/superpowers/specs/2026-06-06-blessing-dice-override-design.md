# 领受赐福 d100 结果锁定设计

## 背景

“领受赐福”当前只是设置页中的本地开关，没有持久化状态，也不会影响投骰。目标是把它变成一个可开启的作弊模式：开启后，在 d100 成功判定类投骰结算前显示结果选项，让玩家跳过随机结果并锁定判定档位。

本次只覆盖 d100 成功判定，不覆盖伤害、SAN 损失等数值骰。数值骰后续可以单独设计为最小值、均值、最大值或自定义值策略，避免把“成功档位”和“数值总和”混在同一个交互里。

## 结果档位

使用项目现有的 `DiceResultType` 六档：

- `crit-success`：大成功
- `extreme-success`：极难成功
- `hard-success`：困难成功
- `success`：成功
- `failure`：失败
- `crit-failure`：大失败

UI 文案沿用现有骰子面板和历史记录的中文标签，确保 `parse-dice-input`、左页历史展示和动画结果能够继续识别。

## 设置与持久化

在设置 store 中新增一个持久化布尔值，例如 `blessingEnabled`，以及对应 setter。`BlessingContent` 不再使用组件内 `useState`，而是读取并更新全局设置。

设置页仍保持现有“赐福开关”结构：一行标签、一个 ON/OFF 胶囊按钮、一个帮助提示。帮助提示改为说明开启后会在 d100 检定结算前显示六档结果锁定按钮。视觉上复用当前设置页的金色分隔线、暗色按钮、低亮度 hover，不引入新风格。

## 结果锁定核心 helper

新增一组纯函数作为所有入口的共同基础：

1. `getBlessingResultOptions(target, sanCheck)`：返回六档中当前目标值下可合法生成的选项，并标明不可用原因。
2. `pickRollForResult(type, target, sanCheck)`：从目标档位反推出一个 d100 点数，并保证 `determineResult(roll, target, sanCheck)` 返回同一档。
3. `buildBlessedCheckResult(...)`：基于原始检定上下文生成新的 `roll`、`resultType`、中文结果行和 `DiceRecord`。
4. `rankOpposedResult(...)`：对抗检定复用现有 `RESULT_RANK` 口径，按玩家侧和对手侧结果重新结算胜负。

反推点数规则：

- 大成功优先使用 `1`。
- 极难成功优先使用 `2..floor(target / 5)`。
- 困难成功优先使用 `floor(target / 5) + 1..floor(target / 2)`。
- 成功优先使用 `floor(target / 2) + 1..target`。
- 失败优先使用 `target + 1..95`，普通检定会避开目标低于 50 时的 `96..100` 大失败区。
- 大失败普通检定优先使用 `100`；SAN 检定可使用 `96..100`，仍优先 `100`。

如果某个档位在当前目标值下没有能被 `determineResult()` 承认的合法点数，UI 禁用该按钮，而不是伪造不一致记录。

## 普通剧情检定

`RightPage` 的普通检定已经生成 `stagingTrigger` 并交给 `OptionResolutionOverlay` 处理，包括：

- 显式目标值检定。
- 从角色卡按普通、困难、极难难度解析目标值的检定。

赐福按钮放在 `OptionResolutionOverlay` 的结果展示区域下方。用户点击某个档位后，overlay 用核心 helper 重建：

- 输入框中的结果行。
- `DiceRecord.roll`。
- `DiceRecord.type`。
- 动画 detail 中的 `roll` 和 `resultType`。
- 后续直接落账、花费幸运、孤注一掷的 UI 状态。

点击“直接落账”时仍走现有 `stashRecord`、`commitToTextarea`、`resolveStore` 流程。也就是说，赐福只是替换 staged 结果，不改变提交管线。

## 手动骰面板

`DicePanel` 在已掷出但未提交时显示“赐福刻印”六档按钮。选择结果后：

- 更新 dice store 当前 staged `finalRoll` 与 `resultType`。
- 更新面板本地显示的最终点数、结果标签和结果文本。
- 重新调用现有 UI 状态推导：失败类结果可继续显示孤注一掷入口，成功类结果不显示。
- `commitNow()` 保持原有逻辑，从 store 当前 staged 状态生成历史记录。

这避免新增一套手动骰提交逻辑，也保证检定历史、成长 tick 和声音/文本反馈继续复用现有路径。

## SAN 检定

SAN 检定里的 d100 成败纳入赐福，SAN 损失表达式的数值骰不纳入本轮。

处理顺序变为：

1. 解析 SAN 成功损失表达式和失败损失表达式。
2. 如果赐福开启，先给出 d100 六档锁定选择。
3. 按锁定后的 `resultType.includes('success')` 选择成功或失败损失表达式。
4. 用 `rollDiceExpr()` 正常掷 SAN 损失数值。
5. 生成结果行与 `DiceRecord`。

因此锁定“大成功 / 极难成功 / 困难成功 / 成功”都会走成功损失分支；锁定“失败 / 大失败”都会走失败损失分支。大失败仍以 `DiceRecord.type` 记录为 `crit-failure`，但不额外改变 SAN 损失表达式。

## 对抗检定

对抗检定采用“玩家侧结果锁定，对手仍正常掷骰”的规则。

流程：

1. 对手照常随机掷骰并计算 `opponentResultType`。
2. 玩家可以选择自己的六档结果。
3. 系统生成与玩家目标值和所选档位一致的玩家 d100 点数。
4. 使用现有 `RESULT_RANK` 规则重新结算 `win / lose / draw`。
5. 记录玩家侧 `DiceRecord.type` 和玩家侧点数，动画 detail 保留对手点数、对手结果和最终胜负。

这属于“极致作弊”但仍保持记录可解释：对手不是被静默篡改，胜负仍能由双方结果推导出来。UI 会把通常能保证胜利的高档位做更明显的金色强调，但不会把低档位强行改成胜利。

战斗入口里的对抗选项保持直接进入战斗面板，不接普通剧情 overlay。实现时可复用同一个玩家侧锁定 helper，但不改变战斗入口的提交/跳转结构。

## 暗骰和数值骰

暗骰不显示赐福按钮。心理学等隐藏检定的语义是玩家不知道真实结果，显示结果锁定会破坏暗骰设计。

伤害骰、SAN 损失骰和其它 `rollDiceExpr()` 表达式不显示六档按钮。本轮不新增数值骰作弊 UI，也不改数值骰记录格式。

## UI 风格

视觉优先匹配现有项目：暗色羊皮纸、皮革底、金色细边、低饱和结果色和 Framer Motion 的轻量淡入。

“赐福刻印”区域只在设置开启且存在可锁定 d100 staged 结果时出现。它应像已有骰子结果卡的附属操作，而不是独立外挂面板：

- 标题短小，例如“赐福刻印”。
- 六个按钮横向或自动换行排列。
- 当前选中结果用金边和低亮金色背景标识。
- 不可用档位降低透明度并禁用。
- 手机端允许两行或三行换行，避免横向溢出。

## 错误处理

- 如果目标值无效或无法为某档生成合法点数，该档禁用。
- 如果 staged 结果已经提交或 pending 已清空，赐福按钮不再响应。
- 如果 settings 关闭，所有入口都回到原行为。
- 如果 SAN 损失表达式解析失败，继续沿用当前逻辑把损失视为 0。
- 如果对抗检定缺少对手目标值，保持现有默认/回退行为，不为赐福新增额外兜底。

## 测试计划

单测：

- 六档结果在不同目标值和 `sanCheck` 下生成合法点数。
- `pickRollForResult()` 的输出再喂给 `determineResult()`，结果必须一致。
- 目标值极低时不可生成的档位会被标记为 disabled。
- 对抗胜负在替换玩家结果后按 `RESULT_RANK` 正确重算。

集成/组件测试：

- 普通剧情检定锁定结果后，输入行、`DiceRecord` 和动画 detail 一致。
- 手动骰锁定结果后，提交历史使用锁定后的点数和类型。
- SAN 检定锁成功时使用成功损失表达式，锁失败或大失败时使用失败损失表达式。
- 关闭设置后不显示赐福按钮。
- 暗骰和伤害骰不显示赐福按钮。

手动验证：

- 打开设置页切换“领受赐福”，刷新后状态保留。
- 走一次普通剧情检定，锁定六档中的不同结果并提交。
- 走一次 SAN 气泡，分别锁成功和失败，确认损失分支不同。
- 走一次对抗检定，确认对手点数仍显示，玩家侧锁定后胜负重算。
- 打开手动骰面板，确认锁定后历史记录和 UI 状态正确。
